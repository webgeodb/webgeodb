# 代码审查检查清单

> 📌 **审查 PR 时必须执行的检查清单**

## 🔍 代码质量

### 类型安全
- [ ] 无 `any` 类型（除非有明确注释说明原因）
- [ ] 接口和类型定义完整
- [ ] 泛型使用正确
- [ ] 类型断言最小化

### 错误处理
- [ ] 所有 async 函数有 try-catch 或错误传播
- [ ] 错误信息清晰有用
- [ ] 避免静默失败（空 catch 块）
- [ ] 正确使用 Error 类型

### 代码风格
- [ ] 遵循命名规范（PascalCase/camelCase）
- [ ] 函数单一职责（< 50 行）
- [ ] 避免深层嵌套（< 4 层）
- [ ] 有意义的变量和函数名

### 性能
- [ ] 无明显性能问题
- [ ] 大数据量考虑优化
- [ ] 避免不必要的计算
- [ ] 合理使用缓存

---

## 🧪 测试质量

### 测试覆盖
- [ ] 新功能有单元测试
- [ ] 边界条件有测试
- [ ] 异常情况有测试
- [ ] 覆盖率 ≥ 80%

### 测试质量
- [ ] 测试命名清晰
- [ ] 测试独立（无依赖）
- [ ] 异步测试正确（使用 await）
- [ ] 测试数据清理完整

### 导入路径检查（重点！）
```typescript
// ✅ 正确 - test/ 目录下的测试文件
import { WebGeoDB } from '../src';

// ❌ 错误
import { WebGeoDB } from '../../src';
```

## ⚠️ 常见问题

### 必须拒绝的 PR
- ❌ 测试未通过
- ❌ 有 TypeScript 错误
- ❌ 导入路径错误
- ❌ 使用 any 过多
- ❌ 无测试覆盖
- ❌ 缺少文档

### 应该请求修改的 PR
- ⚠️ 测试覆盖率 < 80%
- ⚠️ 缺少边界测试
- ⚠️ 性能问题未解决
- ⚠️ 文档不完整
- ⚠️ 代码风格不一致

---

## 📊 审查评分标准

### LGTM（Looks Good To Me）✅
- 所有必须项通过
- 可选项至少 80% 通过
- 无重大问题

### 请求修改 🔄
- 有 1-2 个阻塞性问题
- 或有 3-5 个建议性问题

### 拒绝 ❌
- 有 3+ 个阻塞性问题
- 或测试未通过
- 或有严重安全问题

## 🎯 快速审查流程

1. **快速扫描**（2 分钟）
   - 查看 PR 描述和变更文件
   - 检查导入路径（关键！）
   - 查看测试覆盖率

2. **详细审查**（10-15 分钟）
   - 逐文件审查代码
   - 检查测试完整性
   - 验证文档更新

3. **本地测试**（5 分钟）
   ```bash
   git checkout pr-branch
   pnpm test
   pnpm build
   ```

4. **提供反馈**
   - 使用上述模板
   - 明确指出问题位置
   - 提供改进建议

---

**记住**: 严格但友善的审查是项目质量的保障！
